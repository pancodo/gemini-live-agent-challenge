From Historical Document to AI Documentary: Building an 11-Phase Research Pipeline with Google ADK


This post was written as part of my submission to the Gemini Live Agent Challenge hackathon, organized by Google LLC and administered by Devpost.

---

Upload any historical document — a centuries-old manuscript, a faded diplomatic letter, a scanned page from an ancient text in any language. Forty-five seconds later, watch the first segment of a cinematic documentary about it — narrated by an AI historian you can interrupt and question at any time, illustrated with period-accurate generated images, backed by corroborated research from real web sources.

That is AI Historian. This post covers the engine behind it: an 11-phase pipeline built on Google's Agent Development Kit (ADK) that transforms any historical document into a structured, fact-checked, geographically grounded documentary — streaming progress to the frontend in real time.


The Pipeline at a Glance

The pipeline runs as a SequentialAgent with 11 phases, each one a custom BaseAgent subclass. Phase I handles document analysis — OCR, chunking, and scene selection. Phase II runs parallel web search for each scene. Phase III generates the narration scripts and segment structure. Phase IV uses Gemini's interleaved TEXT+IMAGE capability to produce storyboards. Phase V pre-generates per-beat narration visuals using the same interleaved output. Phase VI classifies each beat as illustration, cinematic, or video to determine its visual generation path. Phase VII is the hallucination firewall — a fact validator that cross-references every claim against research. Phase VIII extracts and geocodes every geographic location in the script. Phase IX produces a director's shot list to ensure visual distinctness across scenes. Phase X runs archival web search for period-accurate visual references. And Phase XI generates the final images with Imagen 3 and video clips with Veo 2.

State flows between agents via session.state — ADK's built-in state sharing mechanism. The entire pipeline is wrapped in a ResumablePipelineAgent that checkpoints to Firestore after each phase, so a crash at Phase X resumes from Phase X, not from the beginning. Every agent receives the same SSE emitter — a channel that streams structured events to the frontend in real time.


Phase I — Document Analysis: OCR to Scene Briefs

The pipeline starts with Google Document AI's OCR processor, which handles 200+ languages — from modern scripts to historical ones. The raw OCR text goes through a deterministic Python chunker — not an LLM call. It splits on page breaks, detected headings, topic shifts, and a 3,200-character hard fallback. Fast and predictable.

Each chunk is then summarized in parallel using Gemini 2.0 Flash, bounded by asyncio.gather with a semaphore of 10 to respect rate limits. The summaries form a DocumentMap — a structured outline of the entire source document.

The interesting part is the Narrative Curator: an ADK Agent running Gemini 2.0 Pro that reads the full document map and makes editorial decisions. It selects 4–6 scenes that form a compelling documentary arc, outputting structured SceneBrief objects — each carrying a scene ID, title, verbatim document excerpt, source chunk IDs, era, location, key entities, and narrative role (opening, climax, resolution, and so on). The Curator also produces a Visual Bible — a style guide that every downstream visual prompt references. It specifies era, palette, composition rules, and what to avoid. This is the creative intelligence of the system: deciding which 5 minutes of a 50-page document become the documentary.


Phase II — Parallel Scene Research: The F-String Trick

Each scene needs independent research to corroborate the specific claims in its source text. The SceneResearchOrchestrator reads the scene briefs from state, fetches raw chunk texts from Firestore, and dynamically builds N research agents — one per scene.

Here is the ADK constraint that shaped the entire architecture: google_search cannot be combined with other tools in the same agent. Every research agent is search-only. No Firestore reads, no LLM post-processing, no tool chaining. Just google_search.

Building the agents dynamically hits a Python/ADK syntax conflict. ADK resolves template variables like {variable_name} in instruction strings from session.state. Python f-strings use the same curly braces. The solution is double-bracing: you write {{scene_{i}_brief}} in the f-string, which produces {scene_0_brief} in the output string, and ADK then resolves that from session.state at runtime. It took a while to get right, but it is clean once you see the pattern.

All agents are wrapped in a ParallelAgent and execute concurrently. Important ADK constraint: ParallelAgent provides no shared state during execution. Each sub-agent writes to its own output key. The downstream aggregator reads all of them.


Phase III — Script Generation

The Script Agent runs on Gemini 2.0 Pro and reads scene briefs plus aggregated research to produce SegmentScript objects — one per scene. Each segment carries a title, a 60–120 second narration script, four Imagen 3 visual descriptions, an optional Veo 2 scene, mood, narrative role, and source citations.

Scripts are written to Firestore immediately. The frontend shows skeleton segment cards that fill in as each segment arrives. One parsing lesson: always handle LLM JSON output defensively. We parse bare arrays, envelope objects, and markdown code fences. The model does not always return the same wrapper.


Phases IV–VI — Gemini's Interleaved TEXT+IMAGE Output

This is the most technically interesting part of the pipeline. Gemini models can produce text and images in a single response — not sequentially, but interleaved. We use this capability in three consecutive phases.


Phase IV — Narrative Director: Storyboarding in One Call

The Narrative Director makes one Gemini call per scene with response modalities set to both TEXT and IMAGE. A single response produces both a creative direction note and a storyboard illustration — the model reasons about visual composition and generates the image in one pass. The response comes back as a list of parts: text parts carry the creative direction, and inline data parts carry the raw image bytes. We upload those bytes to Cloud Storage and the storyboard text feeds downstream phases. If Gemini's safety filter blocks image generation (it occasionally does for historical violence), the phase degrades gracefully and Phase XI falls back to Imagen 3.


Phase V — Beat Illustration: Fast-Path Visuals

Each segment's narration is decomposed into 3–4 dramatic beats. For each beat, Gemini produces an illustration alongside a direction note using the same interleaved TEXT+IMAGE capability. "The fleet assembles at dawn" produces a camera direction note plus a harbor scene image. "The admiral surveys the strait" produces a lighting note plus a close-up portrait. "Cannons fire across the water" produces a composition note plus a wide battle scene.

This is the documentary player's primary visual path. Beat images are available before Phase XI even starts, which is how we hit the under-45-second first-segment target. Scene 0's beats generate first as a fast path; remaining scenes run concurrently. If interleaved generation fails for a specific beat, we fall back to Imagen 3. If that also fails, we skip the beat. The player always has something to show.


Phase VI — Visual Interleave: Cinematic Taxonomy

Not every beat should look the same. The Visual Interleave Agent reads all beats and assigns each one a visual type. Illustration beats keep the Phase V beat image — these are contemplative moments. Cinematic beats get regenerated with Imagen 3 using enriched prompts — these are the dramatic scenes. Video beats get Veo 2 clips — action sequences and transitions.

If a segment has three or more beats, at least one of each type is required — enforced by validation logic, not just the prompt. If Gemini returns invalid types, we fall back to a cycling pattern. This three-way split means the documentary has visual variety without manual art direction. The system decides that a quiet dialogue scene keeps its illustration while a battle gets Imagen 3's cinematic treatment and a ship sailing gets Veo 2 motion.


Phase VII — Fact Validation: The Hallucination Firewall

Before any visual production begins, every narration claim passes through an LLM-judge. The Fact Validator cross-references each sentence in the script against the aggregated research from Phase II.

Every sentence is classified into one of four categories. Supported claims stay exactly as written — "The cathedral was completed in 1386" survives untouched if research confirms it. Unsupported specific claims lose their false precision — if no source confirms "47 ships", the number gets removed. Unsupported plausible claims get softened with hedging language — "He arrived in March" becomes "Historical accounts suggest he arrived in early spring." And non-factual prose stays as-is — "The city held its breath" is rhetoric, not a factual claim, so the validator leaves it alone.

The prompt feeds the model specific hedging phrases: "According to tradition...", "Evidence indicates...", "Historical accounts suggest...". This lets the narration stay cinematic while removing false precision. The validated script only overwrites the original if the segment count matches — a safety check to prevent the validator from accidentally dropping scenes.


Phase VIII — Geographic Mapping

The Geo Location Agent extracts every geographic location mentioned in the documentary script. It uses Gemini 2.0 Flash with a JSON response mode to guarantee structured output — no markdown fences, no prose, just coordinates.

For each segment, the agent produces events (cities, battle sites, regions, each with latitude/longitude, a type marker, and a date context), routes (trade routes, military campaigns, migration paths connecting events in narrative order), and a viewport (calculated center point and zoom level for the map display). The output goes to Firestore and emits SSE events. The frontend renders an interactive map with diamond battle pins, colored route lines, and location markers — the documentary's geography comes alive alongside the narration. Latitude/longitude validation catches swapped coordinates, which is a surprisingly common Gemini mistake.


Phase IX — Visual Storyboard: The Director's Shot List

Before generating final images, we need a plan. The Narrative Visual Planner makes one Gemini 2.5 Flash call across all scenes to produce a Visual Storyboard — a global shot list ensuring no two scenes look the same.

Each scene entry specifies a primary subject (what the image is of, with no duplicates allowed across scenes), a temporal state (is the building newly built or a ruin? does the sculpture have polychrome paint?), a color palette following a warm-to-cool-to-warm emotional arc, an avoid list that references adjacent scenes by name to prevent visual repetition, and four distinct frame concepts.

The temporal state matters more than you might expect. Ancient Greek and Roman sculptures were originally painted in vivid colors — polychrome paint that has since weathered away. Without the temporal state instruction saying "newly built, polychrome paint intact", every image model defaults to white marble. The planner catches these details because it reads the era from the scene brief and reasons about what the subject looked like at the time, not now.


Phase X — Visual Research: Google Search Grounding for Period Accuracy

Before generating images, we research what things actually looked like. The Visual Research Orchestrator runs a 6-stage micro-pipeline per scene using direct Gemini calls with Google Search Grounding. The grounding response includes actual URLs of museum archives, academic papers, and reference images. We fetch these pages, evaluate each source for historical reliability, and merge the accepted details into a Visual Detail Manifest with enriched Imagen 3 prompts.

The manifests include era markers — things that should not appear in the image. These become Imagen 3 negative prompts. If the scene is set in a 16th-century city, the negative prompt prevents modern buildings, electric lights, and automobiles from appearing.


Phase XI — Imagen 3 and Veo 2: Progressive Delivery

The Visual Director reads the manifests and the beat type assignments from Phase VI. Each beat's generation path depends on its assigned type. Illustration beats keep their Phase V images with no regeneration needed. Cinematic beats get Imagen 3 with enriched prompts from the visual research manifest. Video beats get Veo 2 clips. Each Imagen 3 frame gets distinct composition modifiers, lens specifications, film stock references, and era-specific art style anchors. The system maps historical periods to appropriate visual references — Flemish Masters oil painting technique for medieval Europe, David Roberts lithographs for ancient civilizations, and so on. Each era gets its own palette, composition style, and artistic language.

The key to hitting the under-45-second target is progressive delivery. Scene 0 generates first — its frames are created and the SSE event fires before any other scene starts. The user can begin watching while scenes 1 through 5 generate concurrently.

Veo 2 video generation is fire-and-forget. It returns a long-running operation. Here is the gotcha: the operation polling method in the Python GenAI SDK is sync-only. In an async pipeline, you must wrap it with run_in_executor. We poll every 20 seconds, up to 30 times. All Veo 2 polls run concurrently after Imagen 3 completes. When a video finishes, we update Firestore and emit an SSE event. The documentary player hot-swaps the static Ken Burns image for the video clip.


SSE as Content: The Expedition Log

Every agent in the pipeline emits structured SSE events through a shared emitter protocol — pipeline phase changes, agent status updates, source evaluations, segment updates, geographic data, narration beats, and statistics.

The frontend does not show a loading spinner. It renders these events as an Expedition Log — a typewriter-style journal that narrates the research process as it happens. "TRANSLATION & SCAN... Extracting text from 23 pages... FIELD RESEARCH... Researcher 0 searching for 'Roman aqueduct construction techniques 1st century'... CREATIVE DIRECTION... Generating storyboard for Scene 0..." Each log entry types itself character by character with randomized timing. The pipeline has 11 named phases in the Expedition Log, and each new phase triggers a self-drawing gold divider line with an ornament dot.

The 150ms drip buffer is important: parallel agents can emit dozens of events in the same second. The frontend queues them and releases one every 150ms, creating a readable cadence instead of a wall of text appearing at once. This turns pipeline waiting into the first act of the documentary. By the time Scene 0's beats are ready, the user has been engaged watching AI research their document in real time. The loading state is the product.


Lessons and Gotchas

ADK's google_search isolation is not a bug — it is a design choice in the SDK. But it fundamentally shapes your agent architecture. You cannot have an agent that searches the web and then writes to Firestore in the same turn. Plan for it from the start.

ParallelAgent has no shared state. Each sub-agent writes to its own output key. If you need them to coordinate, do it in a downstream agent that reads all the keys.

Pydantic v2 requires ConfigDict with arbitrary_types_allowed set to True when your BaseAgent subclass has non-serializable fields like the SSE emitter or a Firestore client. Without it, Pydantic rejects the field at class definition time.

Always handle LLM JSON output defensively. Gemini sometimes wraps output in markdown fences. Sometimes it returns a bare array. Sometimes it wraps it in an envelope object. Parse all three.

Gemini's interleaved TEXT+IMAGE output is powerful but not guaranteed. Safety filters can block image generation mid-response. Always have a fallback path — in our case, Imagen 3 catches what interleaved output drops.

When you need guaranteed JSON from Gemini, the response_mime_type parameter set to application/json is more reliable than prompt engineering alone. We use it for geo extraction and visual storyboarding.

Imagen 3's negative_prompt parameter works and matters. Without it, you get modern elements leaking into historical scenes. The era markers from visual research directly populate this field.

Temporal state matters for historical accuracy. Image models default to present-day appearance. Ancient sculptures lose their paint, buildings gain modern context. The visual planner explicitly specifies what things looked like at the time, not now.

The Python GenAI SDK's operation polling is synchronous. In an async pipeline, wrap it with run_in_executor. This is not documented prominently.

Resumable pipelines save hours. A crash at Phase XI used to mean re-running OCR, research, and scripting — 3 minutes of wasted compute. Checkpointing to Firestore after each phase means you restart from exactly where you stopped.


The Stack

All three backend services run on Cloud Run — the FastAPI gateway and ADK orchestrator in Python 3.12, the WebSocket relay in Node.js 20. Vertex AI hosts Imagen 3, Veo 2, and all Gemini model calls. Firestore stores session state, agent logs, pipeline checkpoints, segment data, and geographic metadata. Cloud Storage holds uploaded documents, generated images, storyboard frames, and MP4 video clips. Document AI handles multilingual OCR across 200+ languages.

On the AI side, Gemini 2.5 Flash powers the interleaved TEXT+IMAGE storyboards and beat illustrations. Gemini 2.0 Flash handles research, extraction, fact validation, and geo mapping. Gemini 2.0 Pro generates the narration scripts and visual storyboard plans. The entire agent orchestration runs on Google ADK using SequentialAgent, ParallelAgent, and custom BaseAgent subclasses.

Infrastructure is automated with Terraform — a single apply command provisions everything.
